## 1. The gap — what is honestly comparable (cross-team A — Data/legs)
The "חדר מצב — דו"ח ריכוז פוזיציות" card (`telegram_portfolio.py:413` ALGO branch / `telegram_formatters.py:166 fmt_position_card` discretionary branch) and the dashboard position table (`dashboard.py:160 compute_live_portfolio_data` → `live_df['Qty']`, detail expander `dashboard.py:831-835`) show only the **current net `quantity`** per position. Per-position **original entered units / remaining open units / already-realized units / realized-%** are NOT shown on either surface.

**READ-ONLY derivability — CONFIRMED (yes).** `engine_core.split_side_first` (`engine_core.py:509-515`) already establishes the authoritative, validated doctrine inside the byte-locked engine, per campaign group, treating `quantity` as a magnitude (`.abs()`) and `side` as the BUY/SELL discriminant (Phase-C2 / DATA_CONTRACTS "Side and quantity rules"):
- **original entered units** = `Σ |quantity|` over rows where `side=="BUY"` (= `buys_qty`, the engine's own `_c2_buys_qty`).
- **realized units** = `Σ |quantity|` over rows where `side=="SELL"` (= `sells_qty`, the engine's own `_c2_sells_qty`).
- **remaining open units** = `buys_qty − sells_qty` = the engine's existing `net_qty` (`engine_core.py:530`) = the `quantity` field already on every open-position record (`engine_core.py:560`).
- **realized-%** = `sells_qty / buys_qty` (×100), defined ONLY when `buys_qty > 0`; never reported when ambiguous.

The engine emits `quantity`(=net), `initial_qty`/`base_qty`(first-day BUYs only — NOT total-with-add-ons), `add_on_count`, `realized_pnl`, `campaign_id` per record but does **NOT** emit `buys_qty`/`sells_qty`. Therefore the build phase MUST re-derive Σ|BUY| / Σ|SELL| **read-only from the raw trade rows the report builders already hold** — NEVER by changing `engine_core.get_open_positions_campaign` (byte-locked). Both surfaces already hold the raw rows: Telegram `telegram_portfolio.py:234 df = pd.DataFrame(repo.get_all_trades(supabase))` (has `side`/`quantity`/`campaign_id`); dashboard `dashboard.py:289-290 df_sorted` + the per-campaign BUY-row lookup at `dashboard.py:294-300`. The derivation MUST exactly mirror `split_side_first`'s doctrine (`str(side).upper().strip()` eq "BUY"/"SELL"; `quantity.abs()`; the same exact-`trade_id` dedup guard `drop_duplicates(subset=["trade_id"], keep="first")` when a `trade_id` column is present — DATA_CONTRACTS F4) so the derived `net` is byte-identical to the engine's `net_qty`/`quantity` and **no campaign P&L / R / NAV / exposure / campaign-aggregation math is changed**. This is a read-only re-projection of the SAME legs the engine already splits — not a new aggregation.

**The HONEST empty path (cross-team A — INVIOLABLE, AGENTS.md #1).** When the legs are missing or ambiguous, the units-lifecycle figure is rendered as `—` with the explicit Hebrew marker `לא ניתן לאמת` — **NEVER a fabricated number, NEVER a silent zero**. Specifically: (a) no `campaign_id` / no rows for the campaign ⇒ `—`; (b) `buys_qty <= 0` (cannot establish an original base) ⇒ original=`—`, realized-%=`—`; (c) the per-campaign re-derived `net` does not reconcile to the engine's `quantity` for that record (within the existing `0.001` tolerance the engine uses at `engine_core.py:531`) ⇒ the WHOLE units-lifecycle line for that position renders `—` + `לא ניתן לאמת` (a reconciliation mismatch means we cannot trust the leg split — we DO NOT guess); (d) negative-after-split (`sells_qty > buys_qty`, an over-export artifact) ⇒ `—` + `לא ניתן לאמת`. The card's existing `quantity`/`כמות` number is **never replaced** — the lifecycle is an ADDITIVE line; the existing number stays byte-identical (Team B).

## 2. W-R2-1 — pure units-lifecycle helper + single formatter (NEW module, read-only)
New root module (e.g. `position_lifecycle.py`), **not byte-locked**, **zero import of** `engine_core`/`analytics_engine`/`period_data_probe`/`risk_monitor`/Supabase/network. Two pure deterministic functions:
- `compute_units_lifecycle(campaign_rows, *, engine_net_qty)` → a structured dict `{original, remaining, realized, realized_pct, ok: bool, reason}` — re-derives Σ|BUY|/Σ|SELL|/net per the §1 doctrine from the raw rows, applies the §1 honest-empty rules (including the reconciliation check vs `engine_net_qty`), `ok=False` + a `reason` marker on every empty path. Idempotent, no state, no write, never raises (boundary-input discipline; missing/empty/ambiguous ⇒ honest empty, never zero-as-truth).
- `format_units_lifecycle(lc)` → **THE single-source-of-truth display text** — the ONE Hebrew line both surfaces emit, byte-identical for identical input (anti-drift, Phase ALGO-2A §5 doctrine). Honest empty ⇒ the `—`/`לא ניתן לאמת` line; valid ⇒ the original/remaining/realized + realized-% line. Additive only — it returns a NEW line; it never re-formats/reorders/recomputes any existing card number or string.

## 3. Honesty gates (cross-team A/D) — each gates the claim AND forces an honest marker
1. **Reconciliation gate** — re-derived `net` must equal the engine's `quantity` for that record within `0.001`; mismatch ⇒ whole line `—` + `לא ניתן לאמת` (never a partial/guessed lifecycle). 2. **Base gate** — `buys_qty<=0` ⇒ original=`—`, realized-%=`—`. 3. **Sign gate** — `sells_qty>buys_qty` ⇒ `—` + `לא ניתן לאמת`. 4. **No-campaign gate** — absent `campaign_id`/rows ⇒ `—`. 5. **Never-replace gate** — the existing `quantity`/`כמות`/`Qty` value is NEVER overwritten or reordered; the lifecycle is a strictly additive line/column. 6. **Fallback-honesty inheritance** — the line never asserts more certainty than the card's existing price/NAV fallback labels already disclose; it is a units statement only and carries no implied price truth.
