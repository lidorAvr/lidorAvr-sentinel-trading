## Honest disclosure (accuracy over confidence — CLAUDE.md)
The SCOPE §5 anti-drift requirement (a **single formatter**) is fully met and tested: both surfaces render via `format_symbol_divergence`, byte-identical for identical input. However, the **upstream live-aggregate derivation differs per surface**: Telegram (`_build_algo_live_symbol_aggregates`) recomputes the per-trade unit as `total_pnl_usd / original_campaign_risk` and skips campaigns with `original_campaign_risk ≤ 0`; the dashboard reads the engine's existing `Total_Campaign_R` column. These are the same quantity by definition where risk basis is resolved, but the included-campaign population can differ slightly (e.g. campaigns whose R basis the engine resolved differently), so the **per-symbol divergence numbers may differ marginally between the two surfaces**. This is within the literal SCOPE (formatter single-source + byte-identity for the same input), observe-only, explicitly labelled non-truth edge-shape, and never fed into any KPI — so it cannot mislead a decision. **Recommendation:** unify the upstream live-aggregate builder into one shared read-only helper as the first item of **Phase-2b** (where numeric consistency becomes critical because alerting would fire on it). Logged as an open follow-up.
