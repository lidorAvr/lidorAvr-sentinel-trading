## F1 — DATA_CONTRACTS.md example shows the WRONG sign for the founder's actual scenario
- **Where:** `docs/DATA_CONTRACTS.md:504-510` example block shows `"pre_db_realized_pnl_estimate": -495.67`; prose at `:512-513` says "positive = pre-deploy gains, negative = pre-deploy losses". The commit message of `3ac93e8` (OPERATOR DEPLOY section) ALSO uses `-495.67` as the example.
- **Why it matters:** The founder's raw gap on 2026-05-21 was **+$495.67** (NAV $7,856.51 − (deposits $7,500 + DB-realized $-323.88 + open-PnL ≈ $+184) = +$495.67). The math at `telegram_formatters.py:1004` is `adjusted_gap = gap − pre_db_adj`. To neutralise a **+$495.67** raw gap the field must be **+$495.67** (semantic: "pre-deploy GAINS that DB doesn't carry — they were realised before deploy and ARE already in NAV, but not in the expected-equity sum"). The CLI docstring (`scripts/set_pre_db_pnl_estimate.py:25-26`) AND the test `tests/test_meeting_fytd_pre_db_history.py:96-115` (`test_founders_actual_numbers_become_balanced_with_full_disclaimer`) both use the **positive** value and produce `adjusted=0.00` as designed. The data-contract example block, taken at face value, would push adjusted to **+$991.34** (495.67 − (−495.67))  — ESCALATING, not softening. The `min(|raw|,|adj|)` clamp at `:1005` prevents the band from inflating, but the operator's mental model is broken: they'd see in the breakdown "I set the documented value and the gap got WORSE". The chat-log evidence (founder set `=495.67` ~04:10 and got `raw $495.67 → adjusted $0.00`) confirms the CLI/code path is right; the doc is wrong.
- **Severity:** **P1**. Fix-direction: change the example block to `+495.67` (positive). Tighten the prose to spell out the operational rule: "the sign should MATCH the sign of the raw gap you want to neutralise — positive raw gap ⇒ positive disclaimer; negative raw gap ⇒ negative disclaimer." Add the founder's worked example (NAV $7,856 / deposits $7,500 / DB $-323 / open $+184 ⇒ raw $+495 ⇒ set `+495.67`) to anchor the convention. Audit the commit-message text in `3ac93e8` (cannot edit post-hoc but flag in the meeting so future commits don't propagate it).

## F2 — G1-gate reconciliation gap is computed WITHOUT open PnL; display band uses gap WITH open PnL
- **Where:** `adaptive_risk_engine.py:584` `recon_gap = float(nav) − (float(total_deposited) + db_net)` (NO `open_pnl` term) and `report_scheduler.py:317` identical formula vs `telegram_portfolio.py:659-660` and `dashboard.py:404-405` which BOTH include `total_open_pnl` in the expected-equity sum (`acc_size − (deposits + db_net + open)`).
- **Why it matters:** The classifier is fed two different raw gaps depending on the surface, then the SAME `pre_db_realized_pnl_estimate` is subtracted from each — `min(|raw|,|adjusted|)` then bands the result. With non-trivial open PnL (the 21/05 session had ≈$+184) the gate's raw gap is roughly `display_gap + open_pnl` = $+680 vs the display's $+495. After a $+495 disclaimer: gate sees `adjusted = 680 − 495 = $+185`, display sees `adjusted = 495 − 495 = $0`. The display says "Balanced ✅" while the G1 (Clean Data) gate inside `compute_adaptive_risk` might still see "Material Gap" or worse and block a risk raise — or vice-versa with a negative open PnL. The chat-log says the founder DID see Balanced on /portfolio; whether the gate would have permitted a raise at that moment is invisible from the message alone. F-YTD did not create this divergence (it was already present in `build_risk_raise_gate_ctx` pre-commit) but it now propagates through BOTH legs, so the operator's manual estimate has TWO arithmetic effects from one config field.
- **Severity:** **P2**. Fix-direction: either (a) extend `build_risk_raise_gate_ctx` to accept an `open_pnl` kwarg + use it in the gate's gap so the display and gate share ONE classifier input shape, or (b) document the deliberate split in `DATA_CONTRACTS.md` "Data history scope" with the rationale (gate evaluates the STRUCTURAL gap that does not flicker with intraday MtM; display evaluates the AS-OF-NOW gap). Do NOT silently fix — the choice between (a) and (b) is a campaign/risk-evaluation gate semantic and money-affecting per AGENTS.md prime-directive #2.

## F3 — `SENTINEL_CONFIG_PATH` env var override on the CLI does not match the bot's resolver
- **Where:** `scripts/set_pre_db_pnl_estimate.py:65-69` `_resolve_config_path()` returns `Path(_DEFAULT_PATH)` = `./sentinel_config.json` (CWD-relative) when no env var is set, vs `account_state.py:26` `_CONFIG_PATHS = ["/app/sentinel_config.json", "sentinel_config.json"]` (container-path first).
- **Why it matters:** Operator runs `python3 scripts/set_pre_db_pnl_estimate.py 495.67` from `/home/user/sentinel_trading/` on the Orange Pi host. The CLI writes to `/home/user/sentinel_trading/sentinel_config.json`. The bot container, depending on its bind-mount, may be reading `/app/sentinel_config.json` which (if not symlinked / not the same inode as the host file) is unchanged → operator believes they configured the disclaimer; /portfolio still shows "Critical Data Gap"; loop of confusion. The commit message explicitly warns about this ("If the file lives inside the running container instead of the host") but the script does not validate the resolved path matches what the bot would read, nor warn when only one of the two canonical locations exists.
- **Severity:** **P2**. Fix-direction: the CLI should mirror `account_state._CONFIG_PATHS` order on default (try `/app/sentinel_config.json` first, then CWD), and on `--show` print which path it resolved. Operator gets one less foot-gun.

## F4 — `adjustment_applied` is True even when the disclaimer is operationally a no-op
- **Where:** `telegram_formatters.py:1044` `"adjustment_applied": pre_db_adj != 0.0`. Triggered on ANY non-zero field, regardless of whether it actually moved the band.
- **Why it matters:** Set `pre_db_realized_pnl_estimate=200` on a balanced raw gap of $5 ⇒ `min(|5|,|195|)=5` → stays Balanced, but `band_softened=True` at `:1079` ⇒ the message says "after pre-DB history disclaimer ($+200: raw gap $+5 → adjusted $-195)" — the displayed adjusted value flipped sign and grew in magnitude despite the band not moving. Forensically the numbers are real; UX-wise the implication "this disclaimer is doing work" is false. Not a data corruption — a presentation-layer truth claim that's slightly misleading.
- **Severity:** **P3**. Fix-direction: gate `band_softened` on `abs(adjusted_gap) < abs(gap)` (the estimate must demonstrably shrink the gap, not just be non-zero). Wording then matches reality.

## F5 — Breakdown re-computes `gap` from its OWN params instead of trusting `status["gap"]`
- **Where:** `telegram_formatters.py:1153-1154` `expected = float(total_deposited) + float(db_net_pnl) + float(open_pnl)` then `gap = float(nav) − expected`; ignores `status["gap"]` entirely. `adjusted_gap` at `:1161` = `gap − pre_db_estimate` (local gap, not `status["adjusted_gap"]`).
- **Why it matters:** Caller must pass IDENTICAL `nav/total_deposited/db_net_pnl/open_pnl` to both `classify_broker_reconciliation` and `fmt_broker_reconciliation_breakdown`. Any drift (e.g. caller passes the un-rounded NAV to one and the rounded NAV to the other; or `db_net_pnl` from a stale `closed_camps` snapshot vs a fresh `sum(sells.pnl_usd)`) produces a breakdown that contradicts the headline band by a few cents — invariant #8 (no recompute) is technically violated.
- **Severity:** **P3**. Fix-direction: read `gap` and `adjusted_gap` from `status` (the classifier already produced them, already rounded) and assert the locally-computed `expected` matches; on mismatch surface a "values out of sync — refresh" note. Cheap robustness against future drift.

## F6 — DB-net-PnL key drift inside `build_risk_raise_gate_ctx` is undocumented in the contract

## F7 — `pre_db_realized_pnl_estimate` has no audit trail; a future re-sync / fresh deploy silently re-escalates the band
- **Where:** the field lives only in `sentinel_config.json` (gitignored at `.gitignore:3`, per Sprint-28 D-F1 re-verification). The CLI script `scripts/set_pre_db_pnl_estimate.py:88-106` writes the file atomically but does NOT emit an `audit_log` row, and the classifier at `telegram_formatters.py:1003` reads the value per-call without recording it. The four surface callers (`telegram_portfolio.py:262/680/732`, `dashboard.py:595/626`, `risk_monitor.py:1236`, `report_scheduler.py:321`) all use `account_settings.get("pre_db_realized_pnl_estimate", 0) or 0` — so a missing field is indistinguishable from a deliberately-zero one.
- **Why it matters:** If the file is ever regenerated from `sentinel_config.example.json`, restored from a stale backup, or wiped by a Docker-volume reset on the Orange Pi, the field silently reverts to 0 and the band re-escalates from "Balanced ✅" to "Critical Data Gap" with NO breadcrumb explaining why. The CLAUDE.md mandate "if you are not sure whether data is live, stale, cached, estimated, or fallback, say so explicitly" implies the operator should be able to see WHEN the disclaimer was set and to WHAT — currently they cannot. This is also the only "manual operator assertion" in the data contract that does not write to `audit_log`; every other founder-action (RISK-1b lock, RISK-1c backfill, RISK-1d.4 admin correction) goes through `audit_logger.log_action`.
- **Severity:** **P2**. Fix-direction: have the CLI `_set` and `_clear` paths emit an `audit_log` row (action constant e.g. `PRE_DB_PNL_ESTIMATE_SET`/`PRE_DB_PNL_ESTIMATE_CLEARED`, before/after values, operator identity if known). Additive, idempotent, write-only — no read flow changes. Optionally: the disclaimer summary line could include the last-set-date when known ("after pre-DB history disclaimer set 2026-05-21").
